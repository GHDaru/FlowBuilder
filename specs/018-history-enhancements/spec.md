# Feature Specification: Execution History Enhancements

**Feature Branch**: `018-history-enhancements`  
**Created**: 2026-06-04  
**Status**: Draft  
**Input**: Enhanced traceability in History: header context (Firm, Client, Attendant, Flow Name), bulk copy of JSONs, and execution timing tracking (total and per-node).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Rich Execution Header (Priority: P1)

As an IA Evaluator, I want to see the full business context of an execution (Accounting Firm, Client, Attendant, and Flow Name) at the top of the history detail view, so that I can immediately identify the case I am auditing without checking the logs.

**Why this priority**: Essential for efficient auditing of multiple interactions.

**Acceptance Scenarios**:
1. **Given** a selected execution in the History tab, **When** the detail view loads, **Then** the header MUST display the "Contabilidade", "Cliente", "Atendente", and "Fluxo Utilizado".
2. **Given** an execution where metadata extraction failed, **Then** the header MUST display fallback values (e.g., "N/A" or "Não Identificado").

---

### User Story 2 - Bulk JSON Copy (Priority: P2)

As a Flow Creator, I want a single button to copy all JSON responses generated during a specific execution trace, so that I can easily export the full structured result for external analysis or testing.

**Why this priority**: Saves time compared to copying each node's output manually.

**Acceptance Scenarios**:
1. **Given** a trace with multiple LLM nodes, **When** I click the "Copiar Tudo (JSON)" button, **Then** a consolidated JSON object containing all node outputs MUST be copied to my clipboard.
2. **Given** the copy operation, **When** successful, **Then** I MUST see a visual confirmation (e.g., a toast or icon change).

---

### User Story 3 - Execution Timing Metadata (Priority: P1)

As an IA Optimizer, I want to see how long each node took to execute and the total duration of the entire flow, so that I can identify bottlenecks and optimize the prompt/model selection for performance.

**Why this priority**: Critical for production performance monitoring and cost/latency balance.

**Acceptance Scenarios**:
1. **Given** a finished execution, **When** viewing the header, **Then** I MUST see the "Tempo Total" (e.g., "12.4s").
2. **Given** an individual execution step (Accordion), **When** looking at the step metadata, **Then** I MUST see the "Duração" of that specific node (e.g., "1.2s").
3. **Given** the total time, **When** calculated, **Then** it MUST be based on the actual start and end of the aggregate process, not a simple sum of node times (to support future parallel execution).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Backend `Tracking` model MUST include `finished_at` and `total_duration_ms` fields.
- **FR-002**: Backend `TraceLog` model MUST include `duration_ms`.
- **FR-003**: `LangGraphAdapter` MUST record start/end times for each node execution and for the overall process.
- **FR-004**: Frontend `History` detail view MUST render a rich header with business metadata and total timing.
- **FR-005**: Frontend `History` Accordion headers MUST display the individual node duration.
- **FR-006**: Frontend MUST implement a `handleCopyAll` function that merges all `response_json` into a single object and copies it.
- **FR-007**: Metadata (Firm, Client, Attendant) SHOULD be retrieved from the first extraction node's output in the state if possible, or stored in the `Tracking` record.

### Key Entities *(include if feature involves data)*

- **Tracking**: Updated with `finished_at`, `total_duration_ms`, and cached metadata fields.
- **TraceLog**: Updated with `duration_ms`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of new executions record total duration and per-node duration with millisecond precision.
- **SC-002**: Bulk copy generates a valid JSON object containing all node keys from the execution state.

## Assumptions

- We will use standard `datetime.now()` for timing in the backend.
- Metadata (Firm/Client/Attendant) is usually generated by an "Extractor" node. We will look for these keys in the consolidated state.
- Total time will be the difference between the first node's start and the last node's end (or the tracking's internal finish timestamp).
